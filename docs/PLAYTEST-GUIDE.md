# Playtest Guide

This guide is for people who want to help test Live Villages without needing to read code.

You do not need to test everything.

If you notice something odd, confusing, ugly, broken, or incomplete, that is useful feedback.

## What Playtesting Means

Playtesting means:

- playing the mod normally
- trying a few important systems on purpose
- noticing what feels wrong
- reporting bugs or confusing behavior

You are not expected to know the fix.

## What To Test

You can help by checking things like:

- settlement founding
- village names
- Trade Board screens
- villager building behavior
- roads and mileposts
- docks, harbors, and lighthouses
- villager pathing
- textures and models
- whether something is fun, clear, or confusing

## Simple Playtest Loop

If you want a generic test routine, try this:

1. Find or create a settlement area.
2. Place or use a `Trade Board`.
3. Look at the settlement name and basic status.
4. Donate or trade a few items.
5. Place useful infrastructure such as a workstation or anchor.
6. Wait and watch what villagers do.
7. Check whether roads, construction, and UI updates make sense.
8. Report anything broken, weird, ugly, unclear, or unexpectedly slow.

This is meant to be broad on purpose. You do not need to follow it exactly.

## Good Things To Notice

Useful observations include:

- `This broke.`
- `This looked wrong.`
- `This was confusing.`
- `This took too long.`
- `This villager got stuck.`
- `This structure never finished.`
- `This outpost did not feel hostile.`
- `This harbor did not feel connected to water trade.`

All of those are helpful.

## Words To Use In Reports

When writing bug reports, try to mention:

- where you were
- what you placed or clicked
- what happened
- what you expected
- whether it happened again

If you know the right in-game names, use them.

Examples:

- `Trade Board`
- `Surveyor's Table`
- `Portmaster's Anchor`
- `Milepost`
- `village`
- `harbor`
- `outpost`
- `settlement`

If you are not sure which word is right, check [CONTRIBUTOR-GLOSSARY.md](CONTRIBUTOR-GLOSSARY.md).

## Before Reporting A Bug

Before making a new GitHub issue:

1. Check the open issues quickly.
2. If someone already reported the same bug, add a comment there.
3. If not, make a new issue.

The full reporting guide is here:

[BUG-REPORTING.md](BUG-REPORTING.md)

## Discord And GitHub

For this project:

- use Discord for coordination, playtest ideas, and quick questions
- use GitHub issues for bug reports that should be tracked

If Discord is where you first mention a bug, that is fine.

If the bug needs tracking, it should still end up on GitHub.

## You Do Not Need To Be Perfect

A short report is better than no report.

Even something simple like this can help:

```text
I used a Trade Board in a desert village.
The name sounded like a forest village.
I expected a desert-style name.
```
